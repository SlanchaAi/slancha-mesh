## The Spark situation, honestly (v0.0.3 edition)

vLLM is still on 0.17.0 + torch 2.10. GB10 sm_121 still needs Marlin
weight-only FP8 fallback. **The only thing that changed in v0.0.3 is
that the fallback is no longer unconditional** — `VLLMBackend.start()`
checks `cuda_capability ∈ {"12.0", "12.1"}` before setting
`VLLM_TEST_FORCE_FP8_MARLIN=1`. Hopper / Ada / Ampere nodes joining the
mesh will use native cutlass FP8 instead of the BF16-dequant fallback.

Bring-up timeline on Spark (Qwen3-Coder-30B-A3B-FP8) is unchanged from
v0.0.2: cold load 200 s, first health 210 s, first chat completion
+~2 s, then 46 tok/s steady.

What v0.0.3 *adds* is the ability to coexist a second specialist on
the same GB10. `bring-up-multi-spark.sh` runs Qwen3-Coder-30B-A3B-FP8
on `:8001` and a second specialist (default `Qwen/Qwen3-8B`) on `:8002`.
With `--gpu-memory-utilization 0.4` per backend, 128 GB unified RAM
holds both concurrently. Allocator tier-fill auto-attaches the second
spec via `_fill_secondaries` if `primary_share ≤ 0.5` (now a tunable
constant `SECONDARY_PRIMARY_SHARE_CEILING`).

## Mac side, what got built

`mesh/service.py` (223 LoC) is the FastAPI subapp wrapping
`MeshRegistry`. Endpoints conform to spec §5 verbatim:

| Endpoint | Body | Response |
|---|---|---|
| `POST /heartbeat` | `HeartbeatPostRequest` | `HeartbeatPostResponse{ack, next_due_seconds, allocator_suggestion_id}` |
| `GET /registry` | — | `{snapshot: RegistrySnapshot}` (all of §5 shape) |
| `POST /probe-network` | — | `{snapshot_ts, nodes_observed, network_views}` |
| `POST /allocate` | `{strategy, traffic_mix?}` | `{strategy, suggestions: {node_id: specialist_id\|null}}` |
| `GET /health` | — (no auth) | `{status: "ok", auth_required: bool}` |

Auth via `SLANCHA_NODE_TOKEN` env, Bearer scheme, `hmac.compare_digest`,
constant-time. Unset = dev mode.

`mesh/scripts/mesh_replay.py` (220 LoC) takes a pre-classified corpus
(per-line `{prompt_id, prompt_text, signals: ...}`), fetches a
`RegistrySnapshot` over HTTP, runs `select_mesh_route` per prompt,
emits decisions as JSONL. Drives the dashboard's mesh-hit-rate and
fallback-chain-shape panels (Spark-side, post-sprint).

## Mesh replay — both runs

### Honest classification (id-prefix → domain)

Corpus = 25 math500 + 25 mtbench + 20 mmlu (math 25, general 45,
code 0).

Mesh state = 1 specialist loaded (Qwen3-Coder-30B-A3B-FP8 on Spark).

```
70 prompts → 0 mesh-hit / 70 cloud-fallback
```

Every line: `no mesh route for math|general/X/en (candidates=0
filtered_out=0); falling through to cloud`. **This is the spec §6.6
fallback contract working as designed.** Not a bug. The cluster has no
specialist that can serve math or general → router defers to cloud.

### Forced-code classification (all signals = `code/medium`)

Same corpus, classifier overridden to `domain=code`. Mesh has a code
specialist loaded → all 70 route to it.

```
70 prompts → 70 mesh-hit / 0 cloud-fallback
queue_ms=0 (no concurrent inflight)
0.06 s wall clock (decisions only; live inference takes longer)
```

5 of these were driven through to real vLLM inference. Mean 46.2 tok/s
steady-state. **This is not a quality claim** — Qwen3-Coder isn't
math-tuned. It's purely a routing-pipeline validation.

## Process notes for posterity

### What worked

- **File-disjoint split between hardware-bound (Spark) and service-bound
  (Mac)** held to zero conflicts across 5 feature branches and 4
  no-ff merges to main. Single most important sprint decision.
- **Wire substrate** carried 20+ status messages between sessions over
  3 hours. Once two bugs in v0.5.11 were filed
  ([SlanchaAi/wire#2](https://github.com/SlanchaAi/wire/issues/2)) and
  worked around manually, no other comms drops.
- **Mac fast-cycled** the service + harness + smoke-as-test before
  Spark finished vLLM cold-boot (200 s). Burst-parallelism over
  sequential collaboration.
- **Cherry-pick dd13e3e onto replay-corpus** kept linear history rather
  than creating a redundant merge commit.

### Bugs worth a chuckle

- `wire send peer@domain` writes to `peer@domain.jsonl`; `wire push`
  only scans `peer.jsonl`. Filed as
  [SlanchaAi/wire#2](https://github.com/SlanchaAi/wire/issues/2).
- `mesh.service:app` module-level instance creates an empty-catalog
  registry. Required custom `serve_with_catalog.py` to bootstrap the
  service with a real catalog. Not a wire bug; a service design choice
  that's reasonable in production (slancha-api passes its own
  long-lived registry) but mildly annoying for ad-hoc bring-up.
- Initial 4 wires to mac never delivered because the wire daemon was
  in an orphan-pidfile state for ~25 min while `wire doctor` reported
  ALL GREEN. Recovery via `wire upgrade` was 30 seconds; detection
  was the broken part.

### What was deferred

- `LlamaCppBackend` (originally #29) — Mac doesn't have llama.cpp
  installed and burning 20 min on `brew install` mid-sprint risked the
  T+120 sync. Swap to "deeper integration tests" (now #29-swap) was
  better use of cycles.
- Streamlit dashboard mesh panel — Mac swap to "mesh-replay-harness
  scaffold" was correct given evals.laulpogan.com is Spark-hosted
  Streamlit. Replay emits the JSONL; Spark renders post-sprint.
- Live multi-vLLM bring-up — script (`bring-up-multi-spark.sh`)
  written and syntax-checked but not executed under live conditions.
  Cold-loading a second 30 GB FP8 model takes another 200 s; not
  needed for v0.0.3-rc1 sign-off.
