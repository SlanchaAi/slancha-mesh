## The integration seam (spec §7 → live)

v0.0.4 shipped the three pieces (IdleDetector, TrainingPass, ServeDaemon
+ optional `idle_detector` field). v0.0.5 closes the loop:

```
heartbeat tick
  ↓
detector.observe(util, now)
  ↓
if detector.should_start_training()  ← edge fires once
  → daemon spawns TrainingPass thread
  → thread.run(preempt_event=detector.preempt_event)
  ↓
(traffic returns mid-training)
  ↓
heartbeat sees queue_depth > 0
  → daemon.idle_detector.signal_preempt()
  → training thread polls preempt_event each step → yields
  ↓
heartbeat reaps thread (alive=False AND state=TRAINING)
  → detector.finish_training()
  → COOLDOWN (5min lockout against thrash)
  ↓
cooldown elapses → ACTIVE → fresh idle window can accumulate
```

ServeDaemon's new fields (all optional, default None):
- `training_replay_store: TrafficReplayStore | None`
- `training_checkpoint_dir: Path | None`
- `training_kwargs: dict | None`

Both `replay_store` and `checkpoint_dir` must be set for training to fire.
v0.0.4 callers stay unchanged.

## Cross-repo: slancha-local mesh-node registration (#41)

`src/slancha_local/mesh/heartbeat.py` on branch
`feat/mesh-heartbeat-client` (slancha-local repo). Lets any
slancha-local instance register as a mesh node:

```python
loop = MeshHeartbeatLoop(
    registry_url=os.environ.get("SLANCHA_MESH_REGISTRY_URL"),
    node_url=f"http://{settings.bind_host}:{settings.bind_port}",
    friendly_name=socket.gethostname(),
    catalog_fn=lambda: [LoadedSpecialist(...) for m in catalog.all_models],
)
if loop.enabled:
    loop.start()    # daemon thread, posts every 5s
# on shutdown
loop.stop()
```

Wire format matches slancha-mesh spec §5 verbatim. Zero dep on the
slancha-mesh package — payload is a plain dict assembled inline. Same
isolation discipline as `app/router/mesh_client.py` on slancha-api
(walks JSON; no cross-repo pydantic models).

Behavior on slancha-local:
- `SLANCHA_MESH_REGISTRY_URL` UNSET → heartbeat loop never starts;
  no thread spawned; zero HTTP. Default-safe.
- SET → loop posts heartbeat every 5s. HTTP failures log INFO +
  retry next tick. Never raises into the proxy.
- `SLANCHA_NODE_TOKEN` (optional) → sent as Bearer auth.
- `SLANCHA_NODE_ID` (optional) → operator-set stable id; default
  `uuid5(hostname)` (stable across restarts on same machine).

NOT YET WIRED into `proxy/main.py` startup. Module lands + soaks first;
integration into FastAPI lifespan is a follow-up commit.

## Architecture: mesh-of-slancha-locals

The pieces now compose into a coherent local-mesh story:

```
┌──────────────┐         ┌──────────────────┐
│ slancha-api  │ ─uses→  │ select_mesh_first│   pre-existing routing client (v0.0.4)
│ (cloud edge) │         └──────────────────┘
└──────┬───────┘                  │
       │ HTTP /mesh/v1            │ HTTP /registry  + /heartbeat
       ↓                          ↓
┌──────────────────────────────────────┐
│ slancha-mesh.service (FastAPI)       │   v0.0.3 #28 (Mac)
│   /heartbeat /registry /allocate ... │
└────────────┬─────────────────────────┘
             │ posts heartbeat
             ↓
┌────────────────────────────────┐    ┌────────────────────────────────┐
│ slancha-mesh ServeDaemon       │    │ slancha-local proxy            │
│  + IdleDetector (v0.0.4 #34)   │    │  + MeshHeartbeatLoop (v0.0.5)  │
│  + TrainingPass (v0.0.4 #35)   │    │  + classifier client           │
│  + VLLMBackend (v0.0.2)        │    │  + Ollama/llama.cpp/vLLM       │
│  on GB10 Spark                 │    │  on MacBook / Mac mini         │
└────────────────────────────────┘    └────────────────────────────────┘
        ↑                                       ↑
        │ both nodes route domain-matched requests
        │ via /v1/chat/completions per spec §6
        │
```

Each node type is a mesh participant:
- `slancha-mesh ServeDaemon` is the **Spark-side** integration: bare-metal vLLM + idle-fine-tune
- `slancha-local proxy` is the **laptop/Mac-side** integration: multi-backend (Ollama, llama.cpp, vLLM, MLX) under one OpenAI-compat endpoint
- Both register with `slancha-mesh.service` registry; mesh router (in slancha-api or other client) sees a unified `RegistrySnapshot`.

## Process notes

- **Same file-disjoint discipline as v0.0.3/v0.0.4 sprints** — Spark
  touches mesh/{idle,training,replay_store,serve}.py + mesh/scripts/
  + analysis/; Mac touches mesh/dashboard/ + mesh/scripts/install*.sh
  + mesh/deploy/ + protocol-test goldens. Zero merge conflicts
  expected at v0.0.5-rc1 cut.
- **Live smoke unblocked the trust gap** — three v0.0.4 modules tested
  hermetically; v0.0.5 ran them together on actual Spark vLLM with
  real PIDs, real /metrics, real subprocess threading. No mock-vs-real
  surprise.
- **Cross-repo integration without dep coupling** — slancha-local
  doesn't import slancha-mesh; slancha-api doesn't import
  slancha-mesh; slancha-mesh doesn't import either consumer. All
  three communicate via the same JSON wire format documented in
  spec §5. Drift surfaces as test failures, not as runtime
  ImportError.
